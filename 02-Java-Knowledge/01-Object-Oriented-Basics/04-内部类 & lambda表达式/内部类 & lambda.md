**核心感悟**：内部类在逻辑上是“嵌套”，但在内存中是“链表式钩织”。

## 一、 内部类 (Inner Class) —— 内存中的“倒树状链表”

在 JVM 视角下，内部类并不是在大盒子套小盒子，而是通过**引用钩子**连接。

#### 1. 命名与文件 (代号规则)

- **成员内部类**：`Wai$Nei.class`
- **匿名内部类**：`Wai$1.class`（编译器按顺序给没名字的类起代号）。
- **嵌套匿名内部类**：`Wai$1$1.class`（第一个匿名内部类里的匿名内部类）。

#### 2. 隐藏的“脐带”：`this$0` 指针

- **本质**：非静态内部类在 `new` 的瞬间，编译器会自动注入一个 `final` 的成员变量，通常叫 `this$0`。
- **指向**：它指向**堆内存**中离它最近的那个外部类对象的地址。
- **回溯机制**：
  - `this`：指向内部类自己。
  - `Wai.this`：顺着 `this$0` 找妈。
  - 如果是 `this$1`：顺着钩子找姥姥。
  - **类比**：就像一个链表，内部类是头节点，顺着 `this$n` 一直摸到最外层的类（尾节点）。

#### 3. 内存泄漏风险 (Memory Leak)

- **原理**：只要“链表头”（内部类对象）被长生命周期对象（如静态集合、全局监听器）拽住，整条“引用链”上的所有“长辈”（外部类对象）都无法被 GC 回收。



## 二、 变量捕获 (Variable Capture) —— 复印件保护机制

当内部类访问方法里的局部变量时：

1. **矛盾**：栈里的局部变量（原件）随方法结束而销毁，但堆里的内部类对象（复印件）可能还活着。
2. **方案**：内部类在创建时，会将原件的值**拷贝**一份存入自己的成员变量。
3. **强制 final**：为了防止原件和复印件数据不统一导致逻辑混乱，Java 强制要求该局部变量必须是 `final`（或事实上的 `final`），即**不准修改原件**。



## 三、 Lambda 表达式 —— 匿名内部类的“降维打击”

Lambda 是 Java 8 引入的语法糖，旨在解决匿名内部类太“重”的问题。

#### 1. 核心差异对比

| **维度**      | **匿名内部类 (Anonymous)**   | **Lambda 表达式**               |
| ------------- | ---------------------------- | ------------------------------- |
| **底层实现**  | **生成新类** (`$1.class`)    | **动态方法** (`invokedynamic`)  |
| **this 指向** | 指向自己 (`Wai$1`)           | **指向外部类** (没有自己的主权) |
| **内存开销**  | 较重 (每个对象都是新实例)    | 极轻 (通常不持有 `this` 钩子)   |
| **文件表现**  | 硬盘里有对应的 `.class` 文件 | **没有** 对应的独立文件         |

#### 2. 使用前提

- **函数式接口**：有且只有一个抽象方法的接口（如 `Runnable`, `Comparator`）。
